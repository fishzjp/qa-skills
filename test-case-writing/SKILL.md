# 测试用例编写

根据需求文档、技术设计方案或其他输入源，编写系统化的测试用例（markmap 格式）。

## 适用场景

- **有代码仓库**，需基于实际实现编写或审查用例（代码优先，首选场景）
- 给定需求文档/PRD，需要编写测试用例
- 给定技术设计方案，需要编写测试用例
- 给定 API 文档（Swagger/OpenAPI），需要编写接口测试用例
- 给定 Bug 报告，需要编写回归测试用例
- 对已有测试用例进行审查和补充
- 需求变更时，增量更新已有测试用例

## When NOT to Use

- 编写自动化测试代码（单元测试/集成测试/E2E 测试脚本）→ 用 `automated-e2e-testing` skill
- 仅需更新测试数据生成脚本
- 既无代码仓库、也无任何需求/设计文档可参考时

## 工作流程

### 阶段一：输入研读 + 范围界定（代码优先）

从输入源中提取所有可测试点，同时识别平台、角色和测试边界。**代码是最高优先级事实来源**——有代码时以实现为准（测准声明），文档降为对照。

#### 0. 主动索取代码仓库（开工第一件事）

启动测试用例编写前，**主动向用户索要被测项目代码仓库**：

- **索取内容**：① 代码仓库地址（本地路径或远程 URL）；② 被测分支/tag；③ 本次改动范围（PR/MR 链接或 `<base>...<head>` diff 范围，无则默认全量）
- **提问模板**：
  > 🔍 为提升用例准确性，请提供被测项目代码仓库地址与被测分支（若有本次改动的 PR/diff 范围也请给出）。代码将作为功能事实基线，文档作为对照——基于实际实现编写的用例能发现文档与实现的不一致、定位潜在 bug，纯文档模式则无法做到。
- **用户确认无代码 → 降级纯文档模式**：提示「将基于文档编写，准确性受限：无法发现文档与实现的不一致、无法定位潜在 bug、无法核实风险点是否有下游消费」，进入下文「输入源与关注重点」纯文档流程，跳过「代码探索与全面审查」。

> 代码模式与文档模式的差异贯穿后续所有阶段，每个阶段都会标注。

#### 输入源与关注重点

| 输入源 | 重点关注 |
|--------|---------|
| **代码仓库（最高优先级）** | 实际实现逻辑、数据结构、错误处理、与文档偏差、潜在 bug、改动范围（diff） |
| 需求文档/PRD | 功能点列表、边界说明、用户角色、业务规则 |
| 技术设计方案 | 数据模型、接口签名、状态流转、缓存策略、消息格式 |
| API 文档 | 接口参数、必填/选填、返回值、错误码、鉴权方式 |
| Bug 报告 | 复现步骤、根因分析、修复范围（回归用例） |
| FAQ/会议纪要 | 隐含需求、设计决策、边界限制 |

**关键提醒**：FAQ 中的每个问答都是潜在的边界用例；"非目标"声明中的限制需要对应的验证用例。

#### 识别范围

在读输入源时同步记录：
- **平台/端**：管理后台、用户端（App/小程序/H5）、后端 API、第三方回调
- **用户角色**：每个角色能看到什么、能做什么、不能做什么
- **测试边界**：本期范围 vs 不测的 vs 依赖项

#### 受众与执行模型判定（开工必做，决定用例写法）

被测功能有无 UI，直接决定用例正文怎么写。开工时先判定，必要时向用户确认一句：

- **有 UI 的功能**（网页后台、App、小程序、H5 页面）→ 测试工程师**独立执行**。操作步骤写具体的 UI 动作（进入哪个页面、填哪个字段、点哪个按钮）；预期结果写页面上能观察到的现象（出现卡片、提示文案、列表变化）。
- **无 UI 的后端功能**（数据 stage、API 接口、SDK 迁移、定时任务、消息队列消费）→ 测试工程师**无法独立执行**，需写成**「测试-开发协作清单」**：
  - 操作步骤标注「请开发执行」——给开发具体的数据与触发方式；
  - 预期结果标注「请开发反馈」——开发执行后反馈实际值/截图；
  - 验证方式标注「开发查库/查日志反馈」；
  - 三方分工：测试提场景+数据 → 开发执行+查库 → 测试据明确标准判定通过/失败。
  - **开工即问**：向用户确认执行模型（"测试能否独立执行？后端功能通常需开发代跑 + 查库反馈，对吗？"）。

无论哪种模型，**用例正文都必须用业务语言**（见阶段三「正文零代码内部」）。

#### 代码探索与全面审查（代码模式专属，有代码时必做）

有代码仓库时，在文档研读基础上执行以下 4 步，建立项目地图并发现潜在 bug。这步产出后续用例的「代码依据」与缺陷记录基础。

**步骤 1 — 定位与概览**
确认仓库路径/被测分支/改动范围；读 README、路由表、入口文件、配置文件，建立项目地图。

**步骤 2 — 改动盘点**
`git diff <base>...<head> --stat` 列改动文件，逐个分类：
- 🆕 **新建**：新功能主线，全面覆盖
- ✏️ **修改**：回归风险源，重点覆盖原行为是否保持 + 新行为
- 🗑️ **删除**：功能下线，验证下线后无残留入口
- 💀 **死代码**：未挂载路由/未被引用，标记可忽略

产出「改动文件 → 影响模块 → 回归用例」映射（记入附录「改动文件映射」）。

**步骤 3 — 全面代码审查（派 Explore 代理矩阵，始终执行）**

> 按改动文件数/子系统切分，派 1–3 个只读 Explore 代理并行；每个代理聚焦一类发现。代理 prompt 框架示例：
> 「审查 `{仓库路径}` `{分支}` 的 {文件/子系统列表}，针对以下清单逐项查找，每条发现附 `文件:行` 证据并标注确信度：① 潜在 bug；② 文档与实现不一致；③ 死代码/死声明；④ 疑似高风险但可能无下游消费（需核实是否证伪）。只读，不修改。」

审查清单（代理逐项找）：
- **潜在 bug**：null/undefined 未兜底（`.map`/`.length` 炸）、可选链断裂、边界未防护（NaN/负数/空数组/超大值）、状态死锁（改了状态却被门控拦截）、副作用泄漏（全局 state 未 cleanup）、并发竞态、硬编码（如 project_id 固定）、权限遗漏、异步未 await、错误静默吞掉
- **文档 vs 实现不一致**：数据结构字段名/类型、流程顺序、字段是否真提交、默认值
- **死代码/死声明**：未挂载路由的模块、声明未解构的 props、未调用的 API
- **证伪项**：看似高风险实则无下游消费的字段/状态（避免伪用例）

**步骤 4 — 风险分级 + 转化**
- **高风险点 D1-Dn**：按「影响面 × 触发确定性」分级（最高/中高/中/低），每条映射一条回归用例
- **潜在 bug → 缺陷记录 Cx**：编号 + 现象 + `文件:行` 证据 + 处置（主线验证 / 专项验证 / 待实测确认 / 已证伪 / 后端范围）
- **文档偏离**：记入附录「文档偏离对照」
- **死代码**：标注可忽略，不写用例

#### 输入源缺失时的降级策略

- **无代码仓库** → 进入纯文档模式（见上文「0. 主动索取代码仓库」降级提示）：阶段三用例不标「代码依据」，不产出缺陷记录 Cx，审查跳过「代码审查发现」维度
- 只有 PRD，无设计文档 → 不检查"PRD 与技术设计不一致"，审查阶段跳过"技术实现细节"维度
- 只有 API 文档 → 以接口参数和错误码为主要输入源，重点覆盖输入校验和错误处理
- 只有 Bug 报告 → 聚焦回归测试，围绕 Bug 涉及的功能区域展开

### 阶段二：需求澄清

在文档研读完成后，**必须先检查是否存在模糊、不完整或矛盾之处**。发现以下情况时，必须向用户提问确认，不得自行假设：

#### 需要提问的场景

1. **功能描述模糊**：缺少具体业务规则、操作流程或预期行为
2. **边界条件未定义**：如最大/最小值、数量上限、超时时间
3. **状态流转不完整**：状态机有缺失的转换路径
4. **角色权限不明确**：某角色能否执行某操作未说明
5. **PRD 与技术设计不一致**：两边的业务规则/状态流转/操作流程不一致
6. **文档内部矛盾**：同一文档不同位置对同一功能描述不一致
7. **技术设计缺失**：需求中描述的功能在设计方案中无对应实现
8. **异常处理缺失**：未说明错误场景下的预期行为
9. **隐含依赖**：功能 A 依赖功能 B 的某个行为，但该行为未在文档中描述
10. **文档与代码不一致**（代码模式）：数据结构、字段是否提交、流程顺序、默认值等在文档与实现间存在差异

#### 提问格式

```
🔍 需求澄清：

**问题来源**：{文档名} {章节}
**具体问题**：{一句话提问}
**影响模块**：{哪些模块的用例受影响}
**建议选项**：A) ... B) ...（如有）
```

对 PRD 与技术设计不一致，额外列出两边的原文（附章节位置）、差异说明，并询问"测试用例应以哪个文档为准？"

对文档与代码不一致（代码模式），额外列出两边原文（文档章节 + `文件:行`），并询问"以代码为准编写，偏离是否确认为非缺陷？"——默认以代码为准（测准声明）。

#### 输出

- 无需澄清 → 一句话确认"需求已足够清晰"，直接进入阶段三
- 有问题 → 列出问题清单，**等待用户回复后再进入阶段三**
- 后续编写中发现新问题 → 随时补充提问，不硬写

### 阶段三：用例编写

**直接输出最终用例文件，不产出任何中间文件（无 outline、无矩阵表格）。**

使用 markmap 格式（Markdown 标题层级 + 列表）。

#### 测准声明（代码模式必加，文件顶部）

代码模式下，用例文件顶部必须加测准声明：

> **测准声明**：本文件以 `{仓库} {分支}` 实际实现为唯一功能基线；需求/设计文档降为背景对照（偏离见附录）。代码级证据（`文件:行`、缺陷记录、接口对照）统一放文件末尾附录，**不写进用例正文**（见「正文零代码内部」）。

#### 组织方式：状态机驱动

按**状态流转和操作生命周期**组织模块，而非按功能分类。状态机驱动能发现更多遗漏的边界场景。

组织步骤：
1. 从需求/技术方案中提取核心状态机（状态 → 事件 → 新状态）
2. 按状态节点分模块，每个模块覆盖：进入条件、操作行为、转换规则、异常情况
3. 补充横切关注点（权限、通知、数据一致性）为独立模块

#### 格式规范

##### 正文零代码内部（硬约束，最重要）

用例正文是给**测试工程师看和执行**的，必须用业务语言。正文禁出现：

- 代码位置：`文件:行`（如 `set.go:35`、`create.tsx:42`）
- 代码符号：SDK/库的类与函数（如 `srchbase.StringColumn`、`PlatformCli`、`json.Marshal`）
- 错误码与控制流：`resp.Code != 0`、`SetAbort`、`return nil`、`fmt.Errorf`
- 实现术语：`Qualifiers 下推`、`nil 守卫`、`OptionType 分支`

一律改业务语言：`文件:行` → 不出现（落附录）；`resp.Code != 0` → "服务端返回业务错误"；`SetAbort` → "任务进错误文件"；`Qualifiers 下推` → "只返回请求的字段"。

**允许保留**（这些是测试要用的业务/配置事实，不是代码内部）：功能/接口/stage 名（如 `hbase.set`）、配置项 key（如 `fields`）、业务字段名（如 `lr_label_top`）、业务枚举值（如 `DataType=图片`）、可观察的日志原文（如错误信息含 `is required`，作为判定标准时引用）。

> 判定标准：随机挑一条用例，让一个**不读代码**的测试工程师复述"我要验证什么、给什么数据、怎么算通过"。复述不出 → 违反本约束，改写。

##### 详细格式（默认）

**有 UI 功能**用四段式：

```markdown
- **{用例名称}** [P0/P1/P2]
  - 前置条件: {该条特有的前置；无特有时可省略此行}
  - 操作步骤: 1. 进入{页面} 2. 填写{字段} 3. 点击{按钮}
  - 预期结果: {页面上能观察到的明确现象}
```

**无 UI 后端功能**用协作五段式：

```markdown
- **{用例名称}** [P0/P1/P2]
  - 前置条件: {已准备的数据，如测试编号/账号}
  - 操作步骤（请开发执行）: 1. 用{功能名}处理{具体数据} 2. {触发方式}
  - 预期结果（请开发反馈）: {明确唯一的可观察结果}
  - 验证方式: {开发查库/查日志的具体位置与反馈内容}
  - 优先级: P0/P1/P2
```

只有用户明确要求紧凑格式、或信息密度优先（冒烟清单、快速审查）时，才使用紧凑格式：

```markdown
- 操作条件，预期结果 [P0/P1/P2]
```

> 代码模式下 `代码依据: 文件:行` **不再写在每条用例里**；若需溯源，在附录「代码证据清单」统一列表（用例编号 ↔ `文件:行`）。

**前置条件去冗余**：同一子模块下多数用例共享的前置（如"已登录管理员""已创建项目X"），在该子模块标题下方用引用块统一声明一次（`> 前置：...`）；单条用例的「前置条件」行只写该条额外需要的条件，没有额外前置时可省略该行。

**嵌入具体数据**：操作步骤里用具体的测试数据（真实编号、字段名、字段值），不要写占位符 `{xxx}`，让测试能直接照着执行或整包发给开发。

#### 优先级规则（严格执行）

在用例名称行（紧凑格式为整行末尾）标注 `[P0]`/`[P1]`/`[P2]`，不标注时默认 P1。

**P0（冒烟）**— 满足以下**全部**条件：
- 主流程的核心路径
- 失败则整个功能不可用
- **P0 自检**（必须逐条执行）：如果这条用例失败，用户能否完成核心操作？答案为"否"才是 P0

**P1（常规）**：正常功能验证 + 常见异常处理

**P2（边界）**：极端输入、低频场景、性能指标、需特殊环境

#### 用例质量规则

- 一条用例只测一个点，预期结果明确唯一
- 预期结果不能写"或 A 或 B"——应拆分为两条
- 冒烟测试编号前缀为 `SMOKE-{序号}`

#### 可测试性标注

- `[需真机]` → 替代：模拟器 + Mock 数据，标注为 P2
- `[需Mock]` → 替代：写 Mock 脚本并记录在附录中
- `[需专业环境]` → 替代：改为单元测试或标记"需专项测试"

#### Markmap 层级规范

```markdown
# 项目名 测试用例

## 1. 模块名
### 1.1 子模块名
#### 1.1.1 细分类别
- 具体测试点 [P0]
```

附录内容（数据模型、参数定义等）放在文件末尾，不影响思维导图渲染。

#### 模块池（参考）

按项目需要从以下模块中选取，不要求全部覆盖：

基础配置/创建、数据输入/导入、核心业务逻辑、操作/交互、数据处理/转换、搜索/筛选/分页、权限与角色、工作流/审批、通知/消息、文件上传/下载、第三方集成、多轮/链式流程、标签/分类管理、配置管理、边界与异常、安全性、前端展示、兼容性、国际化、性能与并发、数据迁移/升级

> 缺陷记录 Cx、风险点 Dn **不在正文模块池**——它们是代码模式的**附录产出**（见「缺陷记录与风险点产出规范」），放文件末尾，与正文物理隔离。

> 模块拼装的起点模板见 `templates/markmap_template.md`，按项目类型删减组合。

### 阶段四：审查

采用**两层审查**：每写完一个模块立即做轻量审查（4A），全部模块完成后做全局审查（4B）。

#### 4A：模块内审查（每写完一个模块立即执行）

1. **文档段落覆盖**：回到原文档该模块对应的章节，逐段确认是否都有对应用例
2. **代码段落覆盖**（代码模式）：回到代码该模块对应的文件，确认 UI 文案、接口契约、常量、快捷键等是否都有对应用例（文档常在这些细节上过时或错误）
3. **单一职责**：每条用例是否只测一个点，预期结果是否唯一
4. **P0 自检**：重新核对 P0 标注是否通过自检问题

#### 4B：全局审查（全部模块完成后执行）

根据项目规模选择审查强度：

| 强度 | 适用场景 | 执行范围 |
|------|---------|---------|
| 轻量 | 小型项目/紧急场景 | 逐文档溯源 |
| 标准 | 中型项目 | 逐文档溯源 + 多角色视角 |
| 完整 | 大型项目/关键功能 | 逐文档溯源 + 多角色 + 覆盖维度 + 跨模块检查 |

**逐文档溯源**：回到原始输入源，逐章节/段落对照，确保每个可测试点都有对应用例。重点：FAQ 每条至少 1 条边界用例（遗漏高发区）。

**多角色视角**：

| 角色 | 审查重点 | 典型遗漏 |
|------|---------|---------|
| 开发者 | 技术实现细节：数据库约束、缓存行为、异步任务状态；**代码模式下读代码找潜在 bug**，每个潜在 bug 有验证用例；核实风险点是否真有下游消费（证伪伪用例） | 缓存数据变化的场景；潜在 bug 未验证 |
| 产品经理 | 完整操作链路、逆向操作（删除/释放/退回）、配置变更影响 | 只测"领取成功"没测"释放" |
| 运维/管理员 | 数据清理、级联删除、回滚、监控指标 | 正常功能外的清理场景 |
| 测试工程师 | 描述清晰无歧义、每条只测一个点、优先级合理 | 描述模糊如"验证功能正常" |

**覆盖维度**：参考 `references/coverage_checklist.md`，核心维度（功能主流程、输入校验、逆向操作、状态流转、数据一致性、文档隐含需求、代码审查发现）所有项目必检（代码审查发现限代码模式，无代码则跳过）。

**跨模块检查**：重复用例、引用一致、覆盖完整性。

#### 审查输出

**直接在用例文件中补充遗漏和调整**，不产出独立审查报告。只在用例文件顶部或末尾简要记录：

```markdown
## 审查记录
- 审查前：XX 条用例，XX 个模块
- 审查后：XX 条用例，XX 个模块
- 新增：XX 条（{简述补充了哪些场景}）
- 调整：XX 条（{简述调整内容}）
- P0/P1/P2 分布：XX/XX/XX
```

### 缺陷记录与风险点产出规范（代码模式必含）

代码模式的用例文件，除功能用例外，必须含以下产出——**全部放在文件末尾的附录区，与正文用 `---` 分隔**，附录顶部标注"开发技术核查清单（非测试执行项）"：

- **附录：缺陷记录 Cx**：编号（C1、C2…）+ 现象描述 + `文件:行` 证据 + 处置（主线验证 / 专项验证 / 待实测确认 / 已证伪 / 后端范围）
- **附录：高风险点 D1-Dn**：编号（D1、D2…）+ 风险描述 + 等级（最高/中高/中/低）+ `文件:行` 证据 + 覆盖用例编号 + 通过判据
- **附录：代码证据清单**（可选）：用例编号 ↔ `文件:行` 对照，供开发溯源
- **附录：文档偏离对照**：表格列「设计点 / 文档描述 / 实际实现 / 偏离类型」
- **附录：改动文件映射**：表格列「改动文件 / 影响模块 / 回归用例 / 分类（实质·轻量·死代码）」

> 这些代码级内容是给**开发/评审**核查用的，测试工程师不执行、也不应被其干扰。Cx/Dn 中映射的"覆盖用例"指向正文中的业务语言用例。**正文与附录物理隔离是本规范的核心要求**——不要把 Cx/Dn/`文件:行` 当作正文的一个模块（如"模块 N 缺陷记录"）穿插在功能用例之间。

> 纯文档模式（无代码）不产出以上内容，降级时已提示用户准确性受限。

### 阶段五：定稿

定稿前必做**可读性自检**（"给人看、给人执行"的最后把关）：

1. **受众复述**：随机抽 3 条用例，以**不读代码的测试工程师**视角逐条复述"我要验证什么、给谁、给什么数据、怎么算通过"。任何一条复述不出、或含代码术语 → 改写为业务语言。
2. **正文零代码自检**（代码模式）：用 grep 扫描正文（附录 `---` 之前）确认无 `文件:行`、SDK 符号、错误码常量、函数名等代码 token；命中则移入附录。
3. **执行性确认**：每条用例都能被测试独立执行（有 UI），或能整包交给开发协作执行（无 UI）。不能执行又无替代方案的 → 标注可测试性标签或拆分。

根据审查与自检结果调整后的文件即为最终版本。

文件命名：`{项目名}/测试用例_markmap.md`

## 增量更新流程

当需求变更时，增量更新已有测试用例（而非从零重写）。

### 步骤

1. **影响分析**：确定变更影响了哪些模块、哪些用例（直接用例 + 下游依赖 + 状态流转受影响的边 + 数据模型变更影响的查询/校验）。**代码模式**：基于 `git diff` 盘点改动文件，按附录「改动文件映射」的分类定位受影响用例
2. **标记受影响用例**：`[已变更]`、`[已废弃]`、`[需新增]`
3. **更新用例**：新增用追加编号，废弃用 `~~删除线~~` + `[已废弃-原因]`
4. **局部审查**：对变更模块执行 4A 模块内审查 + 涉及变更的跨模块检查
5. **输出变更报告**（追加在审查记录之后）

### 变更报告格式

```markdown
## 测试用例变更报告 — {日期}

- 变更来源：{需求文档版本 / Bug 单号}
- 影响模块：{模块列表}
- 新增用例：XX 条 | 修改用例：XX 条 | 废弃用例：XX 条
```

## Common Mistakes

| 错误 | 后果 | 正确做法 |
|------|------|---------|
| 不做需求澄清直接写用例 | 用例基于错误假设，浪费返工 | 发现模糊立即提问 |
| 一条用例测多个点 | 失败时无法定位原因 | 每条只测一个点，预期结果唯一 |
| 产出中间文件（outline/矩阵/审查报告） | 浪费 token 和时间 | 直接写最终用例，审查结果内联 |
| P0 泛滥或缺失 | 冒烟测试失去筛选意义 | 逐条执行 P0 自检问题 |
| 忽略 FAQ/非目标声明 | 遗漏边界用例 | FAQ 每条至少 1 条用例 |
| 按功能分类而非状态机组织 | 遗漏状态转换边界 | 提取状态机，按转换路径组织 |
| 写出不可执行的用例 | 死用例占空间 | 标注可测试性标签并提供替代方案 |
| 详细格式下每条都重复写共享前置 | 用例冗长、信噪比低 | 共享前置在子模块标题下统一声明，单条只写特有前置 |
| 有代码却不读代码，只靠文档 | 用例基于错误假设（文档常与实现不符） | 有代码必读代码，以实现为准（测准声明） |
| 轻信文档/口头描述，风险点未核实代码 | 写入伪用例（如已证伪的"门控死锁"） | 每个风险点核实是否有下游消费，证伪则降级或删除 |
| 只测功能不查潜在 bug | 遗漏缺陷验证，bug 上线才暴露 | 全面代码审查，潜在 bug 转 Cx 缺陷记录 |
| 用例正文堆代码内部（`文件:行` / SDK 符号 / 错误码常量） | 测试工程师看不懂、无法执行 | 分层：正文纯业务语言，代码证据隔离到文件末尾附录；见「正文零代码内部」 |
| 无 UI 后端功能照搬 UI 用例写"操作步骤" | 测试无法独立执行，步骤悬空 | 判定执行模型；无 UI 写"测试-开发协作清单"，步骤标「请开发执行」 |
